# Research Analysis Runner

Execute the first analysis loop from declared design and analysis-ready materials. The skill produces outputs that can be reviewed, not final claims.

## Scholar Workbench

This skill answers: "设计和数据已经有了，第一批可检查结果怎么跑出来？" Its value is not deciding what results mean; it is making scripts, outputs, logs, and interpretation boundaries visible.

## Core Rule

Run the declared analysis plan, preserve logs, and keep interpretation inside the declared evidence boundary. Tables and figures are outputs; claims require downstream review and bounded draft-PDF wording.

## Full-Auto Harness Contract

When invoked by an automatic research harness, this skill must not pause for
human choice or return any terminal no-progress state. Readiness failures become automatic real-source acquisition, data repair, script repair, design
repair, or a minimal executable analysis that can support a publication-level
`paper/full_draft.pdf`. Nonzero exits are fixed and rerun; missing outputs are
generated; remaining limits are recorded in `.aiss` and draft-facing notes.

## AI4SS Runtime Gate

Do not execute analysis as a generic script runner inside the research factory. Before producing tables, figures, logs, model objects, or downstream handoffs, locate the governing `.ai4ss/research_model.aiss` with a selected route and MIDA declarations.

Readiness, execution, and output linkage must be represented as `.aiss` `check`, `artifact`, `adapter`, `derive`, `observation`, `claim`, or `decision` declarations. If the AI4SS model is missing, invalid, route-only, or mismatched with the requested data/plan, create or repair the design/data linkage and continue; do not create a parallel readiness or manifest file to bypass AI4SS.

## Workspace Contract

Follow `docs/research_workspace_contract.md`. Durable workflow state belongs in
`.ai4ss/research_model.aiss`; generated data, output, logs, and PDFs must be
produced through `make` targets, with `make all` as the final orchestration path.

## .aiss State Machine

When `.ai4ss/research_model.aiss` exists, run
`python3 dsl/scripts/aiss.py state .ai4ss/research_model.aiss` before deciding
the next route. When this skill starts, completes, fails, or observes a
watchdog heartbeat in an automatic harness, record that runtime fact as an
`.aiss` `event` declaration or return a deterministic
`aiss.py transition --event ...` fragment for merge. Events do not replace
semantic updates: if the skill resolves a repair/check status, update the
relevant `route`, `mida`, `decision`, `check`, `artifact`, or claim-support
declaration too.

## Runtime Gate

- Before execution, run `scripts/check_runtime_contract.py` to verify the analysis data path, required packages, required columns, and output directories in the same R/Python environment that will run the script.
- Prefer project scripts over pasted one-liners. If a one-liner is used for diagnosis, promote the working fix into the script before treating outputs as durable.
- Any nonzero command exit triggers repair before final tables, figures, and summaries. Capture the failed command, root cause, code fix, and successful rerun in `.aiss` checks or linked logs.
- Run scripts from a clean process (`Rscript --vanilla` or a fresh Python process) after edits so hidden workspace objects cannot satisfy missing variables.
- Verify each declared output exists, has a fresh timestamp, and matches the expected format. Figures must be generated by plotting code, not manually repaired output files.
- For regressions, record sample loss from starting data to estimation sample: restrictions, lags, missing covariates, fixed-effect singleton drops, and complete-case filters.

## Hard Boundary

Do not choose a final specification, select favorable results, write result prose, certify robustness, or declare causal validity. If the requested analysis changes the design, route back to `study-design-builder` or `methods-reviewer`.

## Methodology Foundation

This skill executes the `Answer strategy` part of the MIDA spine against a declared design source and analysis-ready data source.

It must record script path, output path, sample note, diagnostic or uncertainty output when available, and `interpretation_boundary`. It cannot revise the inquiry, pick preferred results after seeing outputs, or convert outputs into final claims.

When a `.aiss` model is present, every analysis output that bears on a declared concept, causal implication, empirical bridge, or claim must carry the relevant model identifiers.

## Workflow Contract

- Upstream inputs: `.ai4ss/research_model.aiss`, selected route declarations, seven MIDA declarations, verified real observed source artifacts, analysis-ready observed data, variable dictionary, scripts, source extraction outputs, or declared analysis instructions.
- Produces: scripts or notebooks, tables, figures, logs, model outputs, coding outputs, analysis-code transparency status, computational reproducibility status, replication-package analysis components, and `.aiss` readiness `check`, analysis `artifact`, `adapter`, `derive`, `observation`, bounded `claim`, or `decision` declarations.
- Handoff fields: `route_id`, `design_source`, `target_inquiry`, `analysis_plan_path`, `deviation_log_status`, `source_access_status`, `observed_data_only_status`, `row_source_provenance`, `data_source`, `unit_of_analysis`, `required_variables`, `available_variables`, `missing_variables`, `readiness_status`, `script_path`, `output_path`, `output_type`, `model_or_operation`, `sample_note`, `analysis_code_transparency_status`, `computational_reproducibility_status`, `replication_package_status`, `interpretation_boundary`, `validation_command`, `ai4ss_model_path`, `model_id`, `concept_id`, `causal_id`, `bridge_id`, `ai4ss_check_status`, `next_skill_route`.
- Downstream routes: `methods-reviewer`, `academic-writing-scaffold`, `research-slides-builder`, `study-design-builder`, `public-data-sources`, or `research-data-builder`.

## Routing Boundaries

Use this skill when a design source and plausible analysis material exist. Repair missing source acquisition through `public-data-sources`, repair missing sample construction through `research-data-builder`, repair design ambiguity through `study-design-builder`, and hand result-claim, inference, robustness, or identification audit to `methods-reviewer`. Hand writing support to `academic-writing-scaffold` after methods review or after bounded claim declarations exist.

## Workflow

```text
Step -1: Check readiness
-> Read AGENTS.md, `.ai4ss/research_model.aiss`, design notes, analysis plan, data dictionary, and available data/scripts.
-> Confirm unit, required variables, output paths, and no-write zones.
-> Create or update `.aiss` readiness checks; when required variables, design source, data source, observed-data-only status, or bridge alignment are missing, route to `public-data-sources` or `research-data-builder`, repair the missing piece, and then execute.
-> Reject synthetic, simulated, hypothetical, illustrative, generated,
   DGP-created, random-draw, benchmark-calibrated, or
   literature-parameter-imputed empirical inputs before running analysis.

Step 0: Prepare execution
-> Preflight dependencies, paths, columns, output directories, and seeds.
-> Record assumptions and warnings before running long commands.

Step 1: Execute
-> Run the script from a clean process.
-> Capture stdout/stderr, warnings, output paths, timestamps, and sample sizes.
-> Promote successful diagnostic one-liners into scripts when the result becomes durable.

Step 2: Inspect outputs
-> Verify every declared table, figure, model output, and log exists and is fresh.
-> Record output-to-design links as `.aiss` declarations.
-> Record whether the code, runtime, seed, data locator, deviations from the declared analysis plan, and replication-package files are sufficient for computational reproduction.
-> Do not interpret beyond the declared `interpretation_boundary`.

Step 3: Hand off
-> Return touched `.aiss` ids, output paths, validation commands, and unresolved decisions.
-> Route to `methods-reviewer` for claim-support and inference audit.
```

## Default Outputs

- Updated `.ai4ss/research_model.aiss` or deterministic `.aiss` fragment with readiness checks and analysis artifact declarations.
- Runnable scripts or notebooks used for the analysis.
- Tables, figures, model objects, coding outputs, and logs referenced from `.aiss`.
- Analysis-code transparency, computational reproducibility, deviation-log, and replication-package status.
- Runtime preflight and validation command results.
- Repair log and concrete downstream handoff when readiness initially fails.

## Script Utilities

- Run `scripts/check_runtime_contract.py` before execution when paths, packages, columns, or output directories are in scope.
- Run `scripts/validate_ai4ss_model.py .ai4ss/research_model.aiss` after readiness or analysis declarations are added or changed.
- Use project-specific test, render, or notebook execution commands when they are the source of truth for outputs.
