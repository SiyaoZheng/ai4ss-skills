# APSR PDF Harness Agent Instructions

This sandbox receives one idea sentence and must produce `paper/full_draft.pdf`.
Read and follow this file before acting.

## Hard Rules

- Do not block, ask for human judgment, or wait for route/source choices.
- The target is one publication-level draft PDF, not a plan, scaffold, or
  conservative placeholder.
- Empirical data are acquired, not produced. Only real observed public or
  authorized data may enter rows, cells, document codes, events, outcomes,
  treatments, covariates, tables, figures, or estimates.
- Synthetic, simulated, hypothetical, illustrative, generated, DGP-created,
  random-draw, benchmark-calibrated, or literature-parameter-imputed empirical
  data are forbidden. Published parameters may support theory or measurement
  discussion, but cannot become empirical rows or estimates.
- For publication-facing figures, the only final plotting carrier is
  R/ggplot2. Python, Stata, JavaScript, dashboards, screenshots, or interactive
  widgets may support diagnosis or exploration, but final paper figures must be
  generated by reproducible ggplot2 code and exported from R.
  R helper packages may estimate, tidy, compose, label, check, or export, but
  final manuscript figures must remain named ggplot2 objects with explicit
  `ggsave()` calls.
- Paper figures in the same manuscript must share one explicit ggplot2 style
  profile. The profile must cover theme, typography, dimensions, aspect ratios,
  margins, axes, point/line/interval geoms, color and non-color encodings,
  legends, facets, annotations, caption/source-note conventions, export device,
  and filename pattern. Prefer a shared `scripts/figure_style.R` or existing
  project house-style file; record any substantive exceptions.
- If a source route fails, automatically switch source, unit, geography,
  period, measure, or design route. Do not downgrade to synthetic data.
- The final PDF must visibly report source names/URLs, acquisition dates,
  sample construction, measurement choices, analysis commands or reproducibility
  notes, tables or figures with computed values, limitations, and AI disclosure.

## Skill Routing Table

Use this route table deterministically. Each skill must preserve identifiers and
emit `next_skill_route`.

| Skill | Use when | Required input | Required output / next route |
|---|---|---|---|
| `research-starter` | The input is only a one-sentence idea or no selected route exists. | User idea. | Candidate `.aiss` routes, one selected route, assumptions, `next_skill_route: study-design-builder`. |
| `study-design-builder` | The selected route needs MIDA design, estimand/quantity, variables, comparison, or analysis plan. | Selected route in `research_model.aiss`. | Seven MIDA declarations, design choices, source/literature needs, `next_skill_route: public-data-sources` unless verified observed source artifacts already exist. |
| `public-data-sources` | A design needs real observed public/authorized source discovery, access verification, request templates, or bounded acquisition. | MIDA data strategy, unit, period, geography, variables. | Ranked source routes, chosen source, access status, source artifact or request command, provenance, `next_skill_route: research-data-builder` or automatic redesign. |
| `literature-matrix` | Theory, citations, public-source evidence, rival mechanisms, or source-backed contribution claims are needed. | Design, theory needs, seed papers/search terms. | Verified source/span/claim declarations and evidence synthesis, `next_skill_route` to design, methods, writing, or source repair. |
| `research-data-builder` | Acquired real observed source artifacts must become an auditable analysis sample. | Source artifacts/request outputs, data strategy, `research_model.aiss`. | Raw/intermediate/processed data, row provenance, sample checks, `.aiss` source/artifact/empirical declarations, `next_skill_route: research-analysis-runner`. |
| `research-analysis-runner` | Analysis-ready observed data need scripts, tables, figures, logs, or first-pass results. | Processed data, design, analysis plan, required variables. | Rerunnable scripts, tables, figures, model outputs, logs, bounded claims, `next_skill_route: methods-reviewer`. |
| `did-expert` | DID, event study, staggered treatment, panel causal inference, or parallel-trends diagnostics are central. | Panel structure, treatment timing, outcomes, analysis outputs. | Estimator/diagnostic choice, code or bounded identification language, `next_skill_route` to analysis or methods review. |
| `methods-reviewer` | Identification, inference, robustness, reproducibility, result-claim fit, or overclaim risk must be audited. | `.aiss`, scripts, logs, tables, figures, data artifacts, draft claims. | Diagnostic checks, repairs, bounded claim support, `next_skill_route` to analysis/source/data/design repair or writing. |
| `analysis-explainer` | Statistical outputs need social-science explanation, effect-size narration, or limitations. | Tables, models, diagnostics, design context. | Interpretation tied to evidence and limits, `next_skill_route` to writing or methods repair. |
| `latex-tables` | Model or descriptive outputs need publication-ready LaTeX tables. | Model outputs/CSVs/table specs. | LaTeX tables and notes, `next_skill_route` to writing or analysis repair. |
| `top-journal-figures` | Verified empirical outputs need AER/APSR-grade paper figures, ggplot2 code, vector exports, helper-tool triage, forced style consistency, captions, source notes, or visual-integrity checks. | `.aiss`, R scripts, model outputs, figure data, source artifacts, target journal constraints; optional outputs from `fixest`, `did`, `marginaleffects`, `modelsummary`, `cregg`, `binsreg`, `rdrobust`, `sf`, or other R helpers. | R/ggplot2 plotting code, shared style profile, vector-first figures under `output/figures/`, captions/source notes, helper-tool transparency, style-consistency checks, visual-integrity checks, `next_skill_route` to methods review, writing, slides, or analysis/data repair. |
| `academic-writing-scaffold` | Checked evidence must become manuscript sections and the final PDF. | Bounded claims, methods findings, literature, tables, figures. | `paper/full_draft.tex` and `paper/full_draft.pdf`, disclosure and claim boundaries, `next_skill_route: none` only after PDF check passes. |
| `latex-doctor` | LaTeX runtime status is uncertain or a PDF build fails. | TeX source path, build log, or runtime status question. | Runtime diagnosis and missing-tool report, `next_skill_route` to `latex-compile`, writing repair, or none. |
| `latex-compile` | `paper/full_draft.tex` needs rendering or rerendering into `paper/full_draft.pdf`. | Main TeX file and optional engine/output-directory choice. | Built PDF and compile log, `next_skill_route` to writing repair, `latex-doctor`, or none after `make check` passes. |
| `texlive-runtime-installer` | LaTeX support is missing and a managed TeX Live install has been explicitly authorized. | `latex-doctor` missing status and explicit install authorization. | Detected or installed runtime; in this eval prefer existing Docker TeX Live and do not download unless explicitly authorized. |
| `reviewer-response` | Reviewer-style critique or revision routing is needed during repairs. | Comments/findings and affected evidence/design elements. | Revision decisions and repair routes, `next_skill_route` to methods, analysis, source/data, literature, or writing. |

## Execution Contract

Create scripts under `scripts/`, data under `data/`, outputs under `output/`,
and manuscript files under `paper/`. Use `make all` or equivalent explicit
commands. If the OpenAI bundled LaTeX plugin skills are available, use
`latex-compile` for the final TeX-to-PDF build and `latex-doctor` for build
failures; otherwise use the Makefile's `latexmk` target directly. Then run
`make check`. Repair failures before final response.
